# context-engineering 코스 컨텍스트

## 학습 시나리오
당신은 skill-builder를 마치고 스킬 하나를 직접 설계·정리해본 **김신입**이다.
이제 목표는 개인 자동화를 넘어서,
팀 전체가 같은 기준으로 AI를 활용하도록 **워크플로우를 설계**하는 것이다.

## 현재 미션
1. 같은 질문이 레이어에 따라 왜 다른 결과를 만드는지 이해한다.
2. 에이전트 팀과 병렬 작업 구조를 설계한다.
3. MCP 서버를 통해 외부 시스템과 Claude Code를 연결하는 방식을 익힌다.
4. 개인 생산성을 팀 운영 체계로 확장한다.

## 성공 기준
- 프롬프트, `CLAUDE.md`, Plan 모드의 차이를 설명할 수 있다.
- `/tofu-at` 기반 팀 작업 흐름을 말로 설계할 수 있다.
- MCP 서버가 왜 필요한지 비개발자에게 설명할 수 있다.
- Obsidian, Notion, Slack 같은 외부 시스템 연결 시나리오를 설계할 수 있다.
- 팀 차원의 AI 운영 원칙 초안을 만들 수 있다.

## 학습자 프로필
- Claude Code 기본 사용과 스킬 제작 경험이 있다.
- 이제는 "나 혼자 잘 쓰기"보다 팀 적용이 더 중요한 단계다.
- 기술 개념은 어느 정도 익숙하지만, 설계 원칙과 운영 기준은 아직 정리 중이다.
- 목표는 프롬프트를 잘 쓰는 사람이 아니라 **AI 워크플로우 설계자**가 되는 것이다.

## 코스 진행 원칙
- 기능 설명보다 설계 의도를 먼저 본다.
- 도구는 목적을 위해 선택한다.
- 개인 최적화가 팀 혼란을 만들지 않도록 공통 규칙을 강조한다.
- 작은 실험으로 차이를 체감한 뒤 추상화한다.
- skill-builder에서 만든 구조 감각을 팀 운영 언어로 확장한다.

## 김신입의 상황 설정
- 팀 안에서 AI 활용 방식이 사람마다 제각각이다.
- 어떤 사람은 프롬프트만 복사하고, 어떤 사람은 규칙 파일만 믿고, 어떤 사람은 도구 연결을 두려워한다.
- 김신입은 이제 이 혼란을 줄이기 위해 "무엇을 어디에 넣어야 하는지"를 설명하는 사람이 되어야 한다.

## Claude Code에게 기대하는 역할
- 개념 차이를 실험으로 보여주는 코치
- 에이전트 팀 구조를 그려주는 시스템 디자이너
- 외부 도구 연결을 번역해주는 인터페이스 가이드
- 팀 표준을 문서화하게 도와주는 운영 파트너

## 금지된 기대
- 모든 문제를 더 긴 프롬프트로 해결하려 하지 않는다.
- 팀 구조 없이 에이전트를 무작정 많이 늘리지 않는다.
- MCP를 "신기한 연동" 수준에서 소비하지 않는다.
- 개인 취향을 팀 표준으로 바로 강요하지 않는다.

## 코스의 마지막 장면
김신입은 다음 네 가지를 해낸다.
1. 어떤 정보를 어느 레이어에 둬야 할지 판단할 수 있다.
2. 에이전트 팀 작업을 목적에 맞게 설계할 수 있다.
3. 외부 시스템 연동이 필요한 지점을 구분할 수 있다.
4. 팀 전체 AI 워크플로우 초안을 만들 수 있다.

이 코스의 목적은 "고급 기능을 많이 안다"가 아니라,
**"팀이 재사용할 수 있는 AI 운영 구조를 설계한다"** 이다.
