# 1-1 세 개의 레이어 — 프롬프트, `CLAUDE.md`, Plan 모드

## 이 레슨의 위치
이 레슨은 skill-builder를 마친 학습자가
"왜 같은 요청인데 결과가 달라지지?"를 구조적으로 이해하는 첫 단계다.
핵심은 모델이 똑똑한지 아닌지가 아니라,
**어떤 레이어에 어떤 정보를 올렸는가**를 구분하는 것이다.

## 학습 목표
- 프롬프트, `CLAUDE.md`, Plan 모드의 역할 차이를 설명할 수 있다.
- 같은 질문이 레이어에 따라 다른 결과를 낳는 이유를 이해한다.
- 무엇을 어느 레이어에 둘지 판단할 수 있다.

## 세 레이어를 먼저 구분하자
| 레이어 | 역할 | 잘 맞는 정보 |
|--------|------|--------------|
| 프롬프트 | 이번 요청의 즉시 목적 | 지금 당장 할 일, 원하는 결과 |
| `CLAUDE.md` | 프로젝트/사용자 고정 규칙 | 말투, 금지사항, 작업 원칙 |
| Plan 모드 | 구현 전 설계 합의 | 접근 방식, 파일 영향 범위, 작업 순서 |

## 왜 같은 질문도 결과가 달라지나
예를 들어 사용자가 "README 고쳐줘"라고 말했을 때,
어디에 어떤 정보가 있는지에 따라 Claude의 행동은 달라진다.

- 프롬프트만 있으면: 당장 README를 어떻게 바꿀지에 집중한다.
- `CLAUDE.md`가 있으면: 프로젝트 규칙과 스타일을 먼저 따른다.
- Plan 모드까지 쓰면: 바로 수정하지 않고 설계와 영향 범위를 먼저 정리한다.

즉, 결과 차이는 모델의 기분이 아니라 **문맥 계층 차이**에서 나온다.

## 실험
### 실험 1: 프롬프트만 있는 경우
같은 요청을 던지고,
요구사항을 프롬프트에만 적었을 때 어떤 결과가 나오는지 본다.

### 실험 2: `CLAUDE.md` 규칙이 있는 경우
"응답은 짧게", "먼저 계획을 제시" 같은 규칙이 있을 때
출력이 어떻게 달라지는지 비교한다.

### 실험 3: Plan 모드 진입
수정 전에 영향 범위, 접근 방식, 검증 방법을 먼저 설계하면
어떤 종류의 실수를 줄일 수 있는지 관찰한다.

## 판단 기준
### 프롬프트에 둘 것
- 이번 요청만의 목표
- 산출물 형식
- 지금 필요한 맥락

### `CLAUDE.md`에 둘 것
- 자주 반복되는 규칙
- 프로젝트 전체에 적용되는 원칙
- 위험한 행동 금지 규칙

### Plan 모드로 보낼 것
- 구현 전에 합의가 필요한 설계
- 여러 파일에 걸친 변경
- 사용자 선택이 중요한 접근안

## 체크리스트
- [ ] 세 레이어의 차이를 설명할 수 있다.
- [ ] 같은 요청이 왜 다르게 처리되는지 예를 들 수 있다.
- [ ] 고정 규칙과 일회성 요청을 구분할 수 있다.
- [ ] 설계가 필요한 작업에서 Plan 모드의 가치를 이해했다.

## 한 줄 정리
좋은 컨텍스트 엔지니어링은 정보를 많이 넣는 것이 아니라,
**같은 정보라도 어디에 둘지를 정확히 고르는 일**이다.
