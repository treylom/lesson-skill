# 1-3 MCP 이해하기 — Obsidian, Notion, Slack 연동의 기본

## 이 레슨의 위치
이 레슨은 팀 내부 문맥 설계를 마친 뒤,
Claude Code가 외부 시스템과 안전하게 연결되는 방식을 이해하는 단계다.
핵심은 MCP를 "신기한 플러그인"으로 보는 것이 아니라,
**AI가 어떤 도구에 어떤 권한으로 닿는지 설계하는 인터페이스**로 보는 것이다.

## 학습 목표
- MCP 서버의 기본 개념을 설명할 수 있다.
- 왜 단순 프롬프트만으로는 외부 시스템 연동이 어려운지 이해한다.
- Obsidian, Notion, Slack 연동 시나리오를 설계할 수 있다.

## MCP를 한 문장으로 설명하면
MCP는 Claude Code가 외부 시스템의 정보와 기능을
**정해진 도구 인터페이스로 안전하게 다루게 해주는 연결층**이다.

## 왜 필요한가
프롬프트만으로는 아래 한계가 있다.
- 현재 파일 밖의 최신 정보를 직접 가져오기 어렵다.
- 외부 시스템에 구조적으로 쓰기 어렵다.
- 어떤 권한으로 무엇을 읽고 쓰는지 통제하기 어렵다.

MCP는 이런 문제를 해결할 때 쓰인다.

## 대표 시나리오
### Obsidian
- 리서치 노트 읽기
- 정리 문서 업데이트
- 특정 태그나 링크 구조 확인

### Notion
- 데이터베이스 항목 조회
- 문서 상태 동기화
- 협업 문서 초안 반영

### Slack
- 작업 결과 공유
- 특정 채널 공지 초안 작성
- 팀 알림 흐름 연결

## 설계할 때 먼저 물어야 할 질문
1. 왜 이 시스템과 연결해야 하는가?
2. 읽기만 필요한가, 쓰기도 필요한가?
3. 누가 이 결과를 신뢰하고 사용할 것인가?
4. 잘못 썼을 때 위험은 무엇인가?

이 질문 없이 연동을 늘리면,
도구는 많아지지만 운영은 더 혼란스러워진다.

## 좋은 MCP 설계의 기준
- 목적이 분명하다.
- 읽기와 쓰기 권한이 구분된다.
- 실패 시 사람이 검토할 지점이 있다.
- 팀이 이해할 수 있는 운영 규칙이 있다.

## 실습
### 실습 1: 내 팀의 외부 시스템 목록 적기
Obsidian, Notion, Slack 중 무엇을 이미 쓰는지 적는다.
그리고 각 도구에 대해 아래를 적는다.
- 읽고 싶은 정보
- 쓰고 싶은 결과물
- 자동화하면 안 되는 영역

### 실습 2: 연동 우선순위 정하기
세 도구를 한 번에 다 연결하려 하지 말고,
가장 가치가 큰 한 가지 시나리오만 고른다.
예:
- 회의 요약을 Slack에 공유
- 조사 노트를 Obsidian에 저장
- 작업 상태를 Notion에 업데이트

## 한 줄 정리
MCP는 기능 추가 수단이 아니라,
**AI가 외부 세계와 접속하는 방식을 설계하는 운영 계층**이다.
