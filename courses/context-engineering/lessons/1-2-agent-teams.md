# 1-2 에이전트 팀 설계 — `/tofu-at`, Split Pane, 병렬 작업

## 이 레슨의 위치
이 레슨은 개인용 스킬 설계 감각을
팀 단위 실행 구조로 확장하는 단계다.
핵심은 에이전트를 많이 쓰는 것이 아니라,
**누가 어떤 문맥으로 어떤 일을 맡는지 분리하는 것**이다.

## 학습 목표
- 에이전트 팀이 필요한 상황을 구분할 수 있다.
- `/tofu-at` 기반 팀 구성 흐름을 설명할 수 있다.
- Split Pane과 병렬 작업의 장단점을 이해한다.

## 왜 팀이 필요한가
혼자서도 할 수 있는 작업이 많다.
하지만 아래 상황에서는 팀 구성이 이득이 커진다.

- 독립적인 조사 작업이 여러 개일 때
- 문서 작성과 코드 검토를 나눌 수 있을 때
- 한 사람이 모든 맥락을 들고 있기엔 작업이 너무 넓을 때

이때 중요한 것은 "많이 나누기"가 아니라,
**겹치지 않게 나누기**다.

## `/tofu-at`를 보는 관점
`/tofu-at`는 단순 실행 명령이 아니라,
팀 작업을 구조화하는 하나의 운영 방식으로 볼 수 있다.

### 기본 흐름
1. 팀 리드가 목표를 정의한다.
2. 작업을 독립 가능한 단위로 나눈다.
3. 각 에이전트에 역할과 산출물을 명확히 준다.
4. 결과를 다시 중앙에서 통합한다.

## Split Pane이 주는 장점
Split Pane은 여러 흐름을 동시에 관찰하게 해준다.
예를 들어 아래 조합이 가능하다.

- 왼쪽: 메인 구현
- 오른쪽 위: 문서 검토
- 오른쪽 아래: 참고 자료 조사

이 구조는 "작업 동시성"보다도
**맥락 분리** 측면에서 가치가 크다.

## 병렬 작업을 잘못 쓰면 생기는 문제
- 같은 파일을 동시에 고친다.
- 같은 질문을 여러 에이전트가 반복 조사한다.
- 리드가 통합 기준 없이 결과만 많이 받는다.

그래서 병렬화 전에는 항상 아래를 먼저 정해야 한다.
- 산출물이 겹치지 않는가?
- 리뷰 기준이 있는가?
- 누가 최종 결정을 내리는가?

## 실습
### 실습 1: 작업 분해하기
아래 작업을 팀 단위로 나눠본다.
- 강의 코스 작성
- README 개편
- 참고 자료 조사

각 작업에 대해 다음을 적는다.
- 누가 맡는가
- 산출물이 무엇인가
- 다른 작업과 어디서 합쳐지는가

### 실습 2: 병렬화 적합성 판단
아래 질문에 답해본다.
- 이 작업은 서로 다른 파일로 나뉘는가?
- 공유 문맥이 너무 크지 않은가?
- 결과물 품질을 누가 검토하는가?

## 체크리스트
- [ ] 에이전트 팀이 필요한 상황을 설명할 수 있다.
- [ ] 작업을 겹치지 않게 나누는 기준을 안다.
- [ ] Split Pane을 왜 쓰는지 말할 수 있다.
- [ ] 병렬 작업의 실패 패턴을 이해했다.

## 한 줄 정리
좋은 에이전트 팀 운영은 속도의 문제가 아니라,
**문맥과 책임을 깨끗하게 분리하는 설계 문제**다.
