# 1-3 큰 스킬 나누기 — `references/`와 Progressive Disclosure

## 이 레슨의 위치
이 레슨은 첫 스킬 초안을 만든 학습자가,
그 스킬이 커질 때 어떻게 무너지지 않게 관리할지 배우는 단계다.
핵심은 "많이 쓰기"가 아니라 **제때 분리하기**다.

## 학습 목표
- 큰 스킬이 길어질 때 생기는 문제를 설명할 수 있다.
- `references/`로 분리할 대상과 남겨둘 대상을 구분할 수 있다.
- Progressive Disclosure 원칙을 스킬 구조에 적용할 수 있다.

## 왜 분할이 필요한가
처음 만든 스킬은 짧고 선명하다.
하지만 실무에 쓰기 시작하면 보통 이런 것들이 붙는다.

- 입력 예시
- 예외 상황 처리
- 산출물 템플릿
- 품질 점검 체크리스트
- 사용자 유형별 분기

이걸 전부 `SKILL.md`에 넣으면,
핵심 흐름보다 보조 정보가 더 커진다.

## Progressive Disclosure란
Progressive Disclosure는
**지금 꼭 필요한 정보만 먼저 보여주고,
필요한 순간에만 추가 문맥을 읽게 하는 방식**이다.

스킬에서는 보통 이렇게 적용한다.

### `SKILL.md`에 남길 것
- 스킬의 목적
- 핵심 단계 3~6개
- 어떤 순간에 어떤 참고 문서를 읽을지

### `references/`로 보낼 것
- 긴 예시
- 체크리스트
- 템플릿 모음
- 상황별 추가 가이드
- 예외 처리 규칙

## 예시 구조
```text
weekly-report/
├── SKILL.md
└── references/
    ├── input-format.md
    ├── report-template.md
    ├── review-checklist.md
    └── manager-tone-examples.md
```

### 본문에서의 연결 예시
```markdown
## Step 2
입력 자료가 정리되지 않았다면
`Read("references/input-format.md")`를 먼저 확인한다.

## Step 3
초안 작성 시
`Read("references/report-template.md")`의 형식을 따른다.
```

## 실습
### 실습 1: 분리 후보 찾기
"주간보고서 자동 작성" 스킬 초안에서 아래 항목을 찾아본다.
- 단계 설명이 길어지는 부분
- 예시가 필요한 부분
- 품질 점검이 필요한 부분

### 실습 2: `references/` 설계하기
다음 파일을 만든다고 가정하고 역할을 적는다.
- `input-format.md`
- `report-template.md`
- `review-checklist.md`

### 실습 3: 본문 줄이기
`SKILL.md`에서 길게 설명하던 문단을
"언제 어떤 문서를 읽을지" 중심으로 다시 써본다.

## 판단 기준
아래 질문에 "예"가 많을수록 분리할 가치가 크다.

- 이 정보는 모든 실행에서 필요하지 않은가?
- 예시가 길어서 본문 흐름을 끊는가?
- 사용자 유형에 따라 읽을 수도, 안 읽을 수도 있는가?
- 이후 자주 수정될 가능성이 큰가?

## 체크리스트
- [ ] `SKILL.md`와 `references/`의 책임을 다시 나눴다.
- [ ] 본문에는 핵심 흐름만 남겼다.
- [ ] 예시, 템플릿, 체크리스트를 보조 문서로 분리했다.
- [ ] 필요한 순간에만 읽게 하는 구조를 설계했다.

## 한 줄 정리
큰 스킬을 잘 만드는 사람은 내용을 많이 넣는 사람이 아니라,
**무엇을 지금 보여주고 무엇을 나중에 보여줄지 설계하는 사람**이다.
