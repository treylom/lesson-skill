# 1-2 첫 번째 스킬 만들기 — "주간보고서 자동 작성"

## 이 레슨의 위치
이 레슨은 구조 이해를 끝낸 학습자가 처음으로 자기 업무를 스킬로 바꾸는 단계다.
핵심은 거창한 자동화가 아니라, **매주 반복되는 보고 흐름 하나를 안정적으로 재사용**하는 것이다.

## 학습 목표
- 작은 업무 자동화 주제를 스킬로 정의할 수 있다.
- 입력값, 흐름, 산출물을 먼저 설계하는 습관을 익힌다.
- 첫 번째 `SKILL.md` 초안을 직접 만들 수 있다.

## 실습 시나리오
김신입은 매주 금요일마다 아래 일을 반복한다.
1. 한 주 동안의 작업 메모를 모은다.
2. 완료한 일, 이슈, 다음 주 계획을 정리한다.
3. 팀장에게 보낼 주간보고서를 문장으로 다듬는다.

이 과정을 매번 새로 설명하지 않기 위해,
"주간보고서 자동 작성" 스킬을 만든다.

## 먼저 정할 것
스킬을 쓰기 전에 네 가지만 확정한다.

| 항목 | 이번 레슨의 예시 |
|------|------------------|
| 입력 | 메모 파일, 이번 주 완료 항목, 이슈, 다음 주 계획 |
| 출력 | 팀장에게 보낼 주간보고서 초안 |
| 독자 | 팀장 또는 프로젝트 리더 |
| 톤 | 간결하고 업무 중심 |

## 최소 흐름 설계
좋은 첫 스킬은 3~4단계면 충분하다.

1. 입력 자료 확인
2. 핵심 내용 분류
3. 보고서 초안 작성
4. 최종 확인 질문

### 초안 예시
```markdown
---
name: weekly-report
description: 주간보고서 초안 작성
allowed-tools: Read, Write, AskUserQuestion
---

# 주간보고서 자동 작성

## Step 1
입력 자료를 확인한다.

## Step 2
완료 / 이슈 / 다음 주 계획으로 분류한다.

## Step 3
간결한 주간보고서 초안을 작성한다.

## Step 4
빠진 내용이 없는지 사용자에게 확인한다.
```

## 실습
### 실습 1: 내 업무 기준으로 입력값 바꾸기
아래 질문에 답하면서 스킬 요구사항을 좁힌다.
- 주간보고서에 꼭 들어가야 하는 항목은 무엇인가?
- 수치가 필요한가, 서술만 있으면 되는가?
- 보고 대상이 실무자인가, 경영진인가?

### 실습 2: 초안 쓰기
다음을 직접 채운다.
- 스킬 이름
- 설명 한 줄
- 허용 도구
- 3~4단계 실행 흐름

### 실습 3: 결과물 점검
아래 기준으로 초안을 점검한다.
- 이 스킬이 정확히 하나의 일을 하는가?
- 사용자가 무엇을 넣어야 할지 바로 보이는가?
- 산출물이 모호하지 않은가?

## 자주 하는 실수
- 입력이 불분명한데 글쓰기 품질부터 고민한다.
- "나중에 쓸지도 모르는 옵션"을 미리 넣는다.
- 한 스킬 안에서 보고서, 메일, 요약, 번역까지 다 하려 한다.

첫 스킬에서는 욕심을 줄이는 것이 실력이다.

## 체크리스트
- [ ] 입력, 출력, 독자, 톤을 정했다.
- [ ] 3~4단계의 간단한 흐름을 만들었다.
- [ ] 한 스킬이 하나의 핵심 작업만 맡도록 제한했다.
- [ ] 초안을 실제 업무 맥락에 맞게 수정했다.

## 한 줄 정리
첫 스킬의 목표는 완벽한 자동화가 아니라,
**반복 업무를 다시 설명하지 않아도 되는 상태를 만드는 것**이다.
