# 1-1 스킬 해부학 — `SKILL.md`, frontmatter, `references/`

## 이 레슨의 위치
이 레슨은 CC101을 마친 학습자가 처음으로 "사용자"에서 "제작자"로 넘어가는 출발점이다.
이제는 Claude Code를 잘 쓰는 것을 넘어, 같은 작업을 다시 쓸 수 있는 구조로 묶어야 한다.

## 학습 목표
- `SKILL.md`가 스킬의 엔트리포인트라는 점을 이해한다.
- frontmatter에 어떤 정보를 넣는지 설명할 수 있다.
- `references/`가 필요한 시점을 구분할 수 있다.
- 스킬 문서를 500줄 이하로 유지해야 하는 이유를 이해한다.

## 먼저 이해할 것
Claude Code 스킬은 "긴 프롬프트 한 장"이 아니다.
스킬은 **진입점 + 보조 문서 + 실행 흐름**으로 구성된 작은 시스템이다.

### 기본 구조
```text
my-skill/
├── SKILL.md
└── references/
    ├── examples.md
    ├── checklist.md
    └── edge-cases.md
```

### `SKILL.md`의 역할
- 스킬의 이름과 목적을 정의한다.
- 어떤 도구를 쓸지 제한한다.
- 전체 실행 순서를 설명한다.
- 필요한 시점에 `references/` 문서를 읽도록 지시한다.

### frontmatter에서 자주 보는 필드
| 필드 | 의미 |
|------|------|
| `name` | 스킬 이름 |
| `description` | 언제 이 스킬을 써야 하는지 |
| `allowed-tools` | 사용 가능한 도구 목록 |
| `disable-model-invocation` | 모델 호출을 직접 하지 않도록 제한할 때 사용 |

## 왜 500줄 제한이 중요한가
스킬이 길어진다고 품질이 좋아지지는 않는다.
오히려 아래 문제가 생긴다.

- 핵심 흐름보다 예외 상황이 더 눈에 띈다.
- 필요한 정보와 덜 중요한 정보가 섞인다.
- 유지보수할 때 어디를 고쳐야 할지 찾기 어려워진다.
- 모델이 매번 너무 많은 문맥을 한 번에 읽게 된다.

그래서 기본 원칙은 간단하다.

> `SKILL.md`에는 핵심 흐름만 두고,
> 예시·체크리스트·긴 설명은 `references/`로 보낸다.

## `references/`는 언제 쓰나
아래 셋 중 하나라도 해당하면 분할을 고려한다.

1. 예시가 길어져 본문 흐름을 끊을 때
2. 체크리스트가 여러 상황으로 갈라질 때
3. 특정 단계에서만 필요한 설명이 많을 때

이것이 바로 **Progressive Disclosure**다.
처음부터 전부 보여주지 않고, 필요한 순간에 필요한 문서만 읽게 만든다.

## 실습
### 실습 1: 기존 스킬 구조 읽기
아래 항목을 직접 찾아본다.
- `SKILL.md`에 들어가는 최소 필드
- 보조 문서가 있으면 어떤 역할로 나뉘어 있는지
- 본문에 남겨야 할 것과 분리해야 할 것

### 실습 2: "주간보고서 자동 작성" 스킬 뼈대 그리기
다음 질문에 답해본다.
- 이 스킬의 입력값은 무엇인가?
- 최종 산출물은 무엇인가?
- 본문에 있어야 할 단계는 몇 개인가?
- 예시와 체크리스트는 어디로 분리할 것인가?

## 체크리스트
- [ ] `SKILL.md`와 `references/`의 역할을 구분할 수 있다.
- [ ] frontmatter의 핵심 필드를 설명할 수 있다.
- [ ] 500줄 제한이 단순 규칙이 아니라 설계 원칙임을 이해했다.
- [ ] 내 스킬에서 무엇을 본문에 두고 무엇을 분리할지 말할 수 있다.

## 한 줄 정리
좋은 스킬은 "많이 적힌 스킬"이 아니라,
**핵심 흐름이 짧고 보조 문서가 제자리에 있는 스킬**이다.
