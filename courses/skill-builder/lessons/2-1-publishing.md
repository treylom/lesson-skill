# 2-1 스킬 공개하기 — `.claude-plugin/plugin.json`, 마켓플레이스, GitHub 배포

## 이 레슨의 위치
이 레슨은 "내가 쓰는 스킬"을
"팀과 공유할 수 있는 자산"으로 바꾸는 마지막 단계다.
로컬에서 잘 동작하는 것과, 배포 가능한 품질은 다르다.

## 학습 목표
- `.claude-plugin/plugin.json`의 역할을 설명할 수 있다.
- 스킬 공개 전에 확인해야 할 최소 체크리스트를 안다.
- GitHub 배포와 마켓플레이스 등록 흐름을 큰 그림으로 이해한다.

## 왜 배포 단계가 따로 필요한가
로컬에서만 쓰는 스킬은 내가 이해하면 된다.
하지만 공유하는 순간부터는 다른 사람이 아래를 이해할 수 있어야 한다.

- 이 스킬이 언제 필요한가
- 어떻게 설치하는가
- 어떤 입력을 기대하는가
- 버전이 어떻게 관리되는가

이 정보를 묶어주는 것이 `.claude-plugin/plugin.json`이다.

## `plugin.json`에서 보는 핵심 항목
```json
{
  "name": "weekly-report",
  "description": "주간보고서 초안 작성 스킬",
  "version": "1.0.0",
  "author": {
    "name": "김신입"
  }
}
```

실제 필드는 더 있을 수 있지만,
처음에는 아래 네 가지가 가장 중요하다.

| 항목 | 의미 |
|------|------|
| `name` | 플러그인 식별자 |
| `description` | 언제 쓰는지 한 줄 설명 |
| `version` | 배포 버전 |
| `author` | 제작자 정보 |

## 배포 전 체크리스트
### 1. 로컬 사용성 확인
- 설치 경로가 명확한가
- README만 보고 실행할 수 있는가
- 입력 예시가 있는가

### 2. 문서 정리
- 스킬 목적이 한 줄로 설명되는가
- 사용 예시가 있는가
- `references/` 구조가 과하지 않은가

### 3. 공개 가능성 확인
- 팀 내부 정보가 남아 있지 않은가
- 특정 개인만 아는 전제가 없는가
- 민감한 토큰이나 경로가 없는가

## GitHub 배포 흐름
1. 로컬에서 스킬 구조 정리
2. `README.md` 작성 또는 보완
3. `.claude-plugin/plugin.json` 확인
4. Git 저장소에 커밋
5. GitHub에 push
6. 필요하면 마켓플레이스 등록 정보 작성

## 마켓플레이스 등록을 생각할 때
마켓플레이스는 "좋은 아이디어"보다
**처음 보는 사람이 바로 쓸 수 있는지**를 본다.

그래서 아래 질문이 중요하다.
- 설치가 쉬운가?
- 설명이 짧고 선명한가?
- 사용 예시가 바로 떠오르는가?
- 특정 조직 문맥 없이도 가치가 있는가?

## 실습
### 실습 1: 내 스킬 공개 준비도 점검
아래 문장을 직접 채워본다.
- 이 스킬은 누구를 위한 것인가?
- 어떤 반복 업무를 줄여주는가?
- 처음 설치한 사람이 가장 먼저 해볼 일은 무엇인가?

### 실습 2: 배포 체크리스트 만들기
내 스킬 기준으로 아래 항목을 적는다.
- 설치 방법
- 사용 예시
- 주의사항
- 버전 규칙

## 체크리스트
- [ ] `plugin.json`의 핵심 필드를 설명할 수 있다.
- [ ] 공개 전 점검할 항목을 정리했다.
- [ ] GitHub 배포 흐름을 큰 단계로 이해했다.
- [ ] 내 스킬을 다른 사람이 설치할 상황을 상상해봤다.

## 한 줄 정리
배포는 파일을 올리는 일이 아니라,
**다른 사람이 내 스킬을 이해하고 다시 쓸 수 있게 만드는 일**이다.
