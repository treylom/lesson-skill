# skill-builder 코스 컨텍스트

## 학습 시나리오
당신은 CC101을 마치고 첫 자동화 성과를 한 번 만들어본 **김신입**이다.
이제 목표는 Claude Code를 "잘 써보는 사람"에서 끝내지 않고,
팀이 반복해서 쓰는 업무 흐름을 **스킬로 고정**하는 것이다.

## 현재 미션
1. Claude Code 스킬의 기본 구조를 이해한다.
2. "주간보고서 자동 작성" 스킬을 직접 설계한다.
3. 긴 스킬을 `references/`로 분리해 유지보수 가능한 구조를 만든다.
4. `.claude-plugin/plugin.json`과 GitHub 배포 흐름까지 이해한다.

## 성공 기준
- `SKILL.md`의 역할과 frontmatter 필드를 설명할 수 있다.
- `references/`를 언제 쓰는지 판단할 수 있다.
- 주간보고서 자동화 스킬 초안을 스스로 만들 수 있다.
- 플러그인 메타데이터와 배포 체크리스트를 이해한다.
- 다음 단계로 context-engineering 코스를 수강할 준비가 된다.

## 학습자 프로필
- CC101을 통해 Claude Code 기본 사용은 경험했다.
- 파일 읽기, 문서 작성, 간단한 Git 사용은 해봤다.
- 아직 스킬 구조, 배포, 재사용 가능한 워크플로우 설계는 낯설다.
- 목표는 개발자가 되는 것이 아니라 **업무 자동화 설계자**가 되는 것이다.

## 코스 진행 원칙
- 모든 설명은 "실무 자동화" 관점에서 한다.
- 코드보다 먼저 입력값, 산출물, 흐름을 정의한다.
- 한 레슨마다 하나의 핵심 산출물을 남긴다.
- 과한 추상화보다 바로 쓸 수 있는 구조를 우선한다.
- CC101에서 익힌 기본기를 스킬 설계 언어로 연결한다.

## 김신입의 상황 설정
- 팀장은 "좋은 프롬프트"보다 "반복 가능한 방식"을 원한다.
- 동료들이 같은 작업을 매번 새로 설명하는 데 시간이 많이 든다.
- 그래서 김신입은 자주 반복되는 업무를 스킬로 묶어 팀의 공통 자산으로 만들려 한다.

## Claude Code에게 기대하는 역할
- 스킬 구조를 해부해주는 기술 코치
- 초안을 빠르게 잡아주는 페어 디자이너
- 너무 큰 스킬을 적절히 나누게 해주는 편집자
- 배포 전에 빠진 것을 점검해주는 체크리스트 파트너

## 금지된 기대
- 처음부터 완벽한 범용 스킬을 만들려 하지 않는다.
- 아직 쓰지 않을 옵션과 설정을 미리 넣지 않는다.
- 스킬이 길어진다고 무조건 한 파일에 몰아넣지 않는다.
- 마켓플레이스 배포 전에 로컬 사용성을 검증하지 않는다.

## 코스의 마지막 장면
김신입은 다음 네 가지를 해낸다.
1. 좋은 스킬의 구조를 설명할 수 있다.
2. 작은 업무 자동화 스킬을 직접 만들 수 있다.
3. 긴 스킬을 분할해 관리할 수 있다.
4. 스킬을 팀과 공유할 준비를 마친다.

이 코스의 목적은 "스킬 문법을 외운다"가 아니라,
**"반복 업무를 재사용 가능한 Claude Code 스킬로 바꾼다"** 이다.
