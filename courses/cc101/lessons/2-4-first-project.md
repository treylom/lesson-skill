# 2-4: 첫 프로젝트 — practice-data 분석 → 보고서.md → GitHub
## 학습 목표
- 지금까지 배운 내용을 한 흐름으로 연결한다.
- `practice-data/`를 분석해 첫 보고서 초안을 만든다.
- Claude Code와 함께 계획, 분석, 정리, 저장의 사이클을 경험한다.
- Git과 GitHub를 활용해 결과를 정리하는 첫 프로젝트를 완성한다.

## 사전 준비
- 1모듈과 2모듈 주요 레슨 완료
- `practice-data/` 폴더 사용 가능
- 보고서를 저장할 작업 폴더 준비
- Git 기본 개념 이해
- "완벽한 보고서"보다 "한 사이클 완주"가 목표라는 합의

## 진행
### 도입 (5분)
- 드디어 첫 프로젝트다.
- 지금까지는 각 기능을 따로 연습했다.
- 이제는 그것들을 하나로 묶어본다.
- 오늘 김신입의 목표는 분명하다.
- `practice-data`를 읽는다.
- 분석 포인트를 뽑는다.
- `보고서.md` 초안을 만든다.
- 그리고 그 결과를 Git과 GitHub로 정리한다.
- 이 프로젝트가 중요한 이유는 단순하다.
- "Claude Code를 배웠다"가 아니라,
- **"Claude Code로 실제 업무 한 바퀴를 돌렸다"**가 되기 때문이다.
- 완성도의 기준은 전문가 보고서가 아니다.
- 한 번 끝까지 해보는 경험이 핵심이다.

#### 오늘의 전체 흐름
1. 폴더와 데이터 파악
2. 계획 세우기
3. 분석 질문 던지기
4. 보고서 초안 만들기
5. Git으로 저장하기
6. GitHub로 올리기

#### 체크 포인트
- 수강생이 오늘 해야 할 최종 결과물을 알고 있는가?
- 중간에 막혀도 한 단계씩 되돌아가면 된다는 점을 이해하는가?
- 오늘은 완성도가 아니라 연결 경험이 중요하다는 데 동의하는가?

### 실습 1 (15분)
#### 실습 목표
- 프로젝트 전체 계획을 세운다.
- 데이터 분석 질문을 정리한다.
- 보고서 초안 구조를 만든다.

#### 따라 하기
1. Claude Code에게 아래처럼 요청한다.

```text
저는 제조업 회사의 신입 AI 담당자 김신입이에요. practice-data의 sales_data.csv와 meeting_notes.md를 바탕으로 첫 분석 보고서를 만들고 싶어요. 바로 작성하지 말고, 먼저 전체 작업을 6단계 계획으로 보여주세요.
```

2. 계획이 나오면 검토한다.
3. 이어서 분석 질문을 뽑는다.

```text
좋아요. 이 프로젝트에서 꼭 확인해야 할 분석 질문 5개를 우선순위대로 정리해주세요.
```

4. 이제 보고서 틀을 만든다.

```text
보고서.md 초안 구조를 만들어주세요. 제목, 배경, 핵심 인사이트, 원인 가설, 제안 액션, 다음 단계 섹션으로 구성해주세요.
```

5. 마지막으로 오늘의 작업 순서를 더 짧게 만든다.

```text
제가 지금 바로 따라할 수 있게 오늘 할 일을 체크리스트로 바꿔주세요.
```

#### 기대 결과
- 프로젝트가 막연한 분석이 아니라 단계 있는 흐름으로 보인다.
- 분석 질문이 먼저 정리되어 보고서 방향이 선다.
- 빈 문서가 아니라 구조가 있는 보고서 초안이 생긴다.

#### 강사 설명 포인트
- "좋은 프로젝트는 좋은 질문에서 시작합니다."
- "보고서는 마지막에 쓰지만, 구조는 초반에 잡아두는 게 편합니다."

### 실습 2 (15분)
#### 실습 목표
- 데이터와 회의록을 연결해 보고서 초안을 만든다.
- 보고서 파일을 저장하고 Git 흐름까지 연결한다.
- 첫 프로젝트를 닫는 경험을 한다.

#### 따라 하기
1. 데이터 분석을 요청한다.

```text
practice-data/sales_data.csv를 읽고 월별, 제품별, 지역별 관점에서 핵심 인사이트 3개를 정리해주세요.
```

2. 이어서 회의록 연결을 요청한다.

```text
practice-data/meeting_notes.md를 읽고 방금 인사이트와 연결될 수 있는 운영 이슈나 현장 맥락을 정리해주세요.
```

3. 이제 보고서 초안을 요청한다.

```text
위 내용을 바탕으로 보고서.md 초안을 작성해주세요. 경영진에게 공유할 수 있는 톤으로, 너무 딱딱하지 않게 써주세요.
```

4. 초안이 나오면 수정 요청을 한다.

```text
좋아요. 내용을 더 짧고 명확하게 다듬고, 마지막에 실행 제안 3가지를 bullet로 정리해주세요.
```

5. 이제 Git 흐름을 연결한다.

```text
이제 이 결과를 Git으로 저장하고 GitHub에 올리려면 제가 어떤 순서로 하면 되는지 초보자용으로 다시 안내해주세요. 특히 "저장해줘"와 "올려줘" 관점으로 설명해주세요.
```

6. 마지막으로 회고 질문을 한다.

```text
오늘 제가 Claude Code로 해낸 일을 5줄로 정리해주세요. 그리고 다음 프로젝트에서 더 잘하려면 무엇을 개선하면 좋을지 3가지만 알려주세요.
```

#### 기대 결과
- 보고서 초안이 실제 문서 형태로 나온다.
- 데이터와 회의록을 연결해 해석하는 경험을 한다.
- Git/GitHub까지 이어지며 프로젝트 한 사이클을 완주한다.

#### 추천 산출물 예시
- `보고서.md`
- `README.md` 간단 설명
- Git 커밋 1회 이상
- 가능하면 GitHub 원격 저장소 연결

#### 강사 설명 포인트
- "오늘의 승리는 완벽한 분석이 아니라, 끝까지 연결한 경험입니다."
- "다음부터는 이 틀을 다른 업무에도 재사용할 수 있습니다."

### 마무리 (5분)
- 축하한다.
- 김신입은 이제 Claude Code를 단순 체험한 것이 아니라,
- 실제 업무 흐름 안에서 사용해봤다.
- 폴더를 탐색했고,
- 맥락을 줬고,
- 계획을 세웠고,
- 데이터를 분석했고,
- 보고서를 만들었고,
- 저장과 공유까지 연결했다.
- 이것이 바로 첫 프로젝트 완주다.
- 이제 중요한 것은 더 어려운 기술보다,
- 이 흐름을 본인 업무에 다시 적용하는 것이다.

#### 오늘의 한 줄 정리
- "작은 프로젝트 하나를 끝내면, 다음 업무는 훨씬 덜 막막해진다."

#### 실습 후 바로 해볼 것
1. 오늘 만든 보고서 초안의 제목을 3가지 버전으로 바꿔본다.
2. Claude Code에게 "이 흐름을 내 실제 업무에 적용하면 어떤 프로젝트가 좋을까요?"라고 묻는다.
3. 다음 주 안에 반복해볼 30분짜리 미니 프로젝트를 추천받는다.

## 트러블슈팅
### 증상 1: 프로젝트가 너무 크게 느껴진다
- 해결: 오늘의 산출물을 하나로 줄인다.
- 예: 보고서 초안까지만
- 예: 분석 질문 정리까지만

### 증상 2: 데이터 해석이 자신 없다
- 해결: 인사이트를 완성하려 하지 말고 "눈에 띄는 점"부터 적는다.

```text
정답처럼 말하지 말고, 관찰한 점과 가설을 나눠서 써주세요.
```

### 증상 3: 보고서 문장이 너무 딱딱하다
- 해결: 대상 독자를 지정한다.

```text
팀장에게 공유하는 실무 보고 톤으로 다시 써주세요.
```

### 증상 4: Git 단계에서 다시 무서워진다
- 해결: 저장과 올리기를 구분해서 다시 묻는다.

```text
지금은 저장 단계만 다시 설명해주세요.
```

### 증상 5: GitHub 연결까지 못 갔다
- 해결: 괜찮다.
- 오늘의 핵심은 전체 흐름 이해다.
- 로컬 저장까지 해도 절반 이상 성공이다.
