# 2-1: 플러그인 설치 — gptaku + tofukyung 플러그인 붙이기
## 학습 목표
- 플러그인을 "기능 확장팩"처럼 이해한다.
- Claude Code가 기본 기능 외에 더 많은 도움을 받을 수 있다는 점을 배운다.
- gptaku와 tofukyung 플러그인을 설치하거나 연결하는 흐름을 이해한다.
- 설치 전 확인, 설치 후 확인이라는 안전한 습관을 익힌다.

## 사전 준비
- 1모듈 완료
- Claude Code 기본 사용 경험
- "플러그인"이 낯설어도 괜찮다는 마음
- 설치는 한 번에 하나씩 한다는 원칙
- 문제가 생기면 되돌아가 확인할 준비

## 진행
### 도입 (5분)
- 스마트폰에 앱을 깔면 기능이 늘어나듯, Claude Code에도 확장 기능이 붙을 수 있다.
- 여기서는 그것을 플러그인이라고 생각하면 된다.
- 기본 Claude Code가 만능은 아니다.
- 어떤 작업은 외부 도구나 추가 기능과 연결될 때 훨씬 편해진다.
- 오늘은 gptaku와 tofukyung 플러그인을 예시로 본다.
- 중요한 것은 설치 명령을 외우는 것이 아니다.
- 더 중요한 것은 다음 세 가지다.
- 하나, 지금 무엇을 설치하려는지 안다.
- 둘, 설치 전후 상태를 확인한다.
- 셋, 이상하면 바로 멈추고 다시 묻는다.
- 초보자는 "시키는 대로 치는 사람"이 아니라 "무엇을 붙이는지 아는 사람"이 되어야 한다.

#### 쉬운 비유
- 플러그인은 드라이버 세트에 비트 하나를 더 끼우는 일과 비슷하다.
- 본체는 같지만 할 수 있는 일이 늘어난다.

#### 체크 포인트
- 수강생이 플러그인의 개념을 이해했는가?
- 설치 전후 확인의 필요성을 이해했는가?
- 이름이 비슷한 플러그인을 헷갈리지 않겠다는 경계심이 생겼는가?

### 실습 1 (15분)
#### 실습 목표
- 현재 Claude Code 환경에서 플러그인 관련 정보를 확인한다.
- 무엇이 이미 설치되어 있는지 묻는 습관을 만든다.
- gptaku와 tofukyung 플러그인의 역할을 쉽게 설명받는다.

#### 따라 하기
1. Claude Code에게 아래처럼 묻는다.

```text
지금 제 환경에서 플러그인이나 추가 기능 상태를 점검할 때 무엇을 확인하면 좋은지 초보자용 체크리스트로 알려주세요.
```

2. 이어서 역할 이해부터 한다.

```text
gptaku 플러그인과 tofukyung 플러그인이 각각 어떤 상황에서 유용한지 비개발자 눈높이로 설명해주세요.
```

3. 설명을 읽고, 실제 설치 전에 아래처럼 요청한다.

```text
설치 전에 확인해야 할 것과 설치 후 정상 동작을 확인하는 방법을 나눠서 알려주세요.
```

4. 메모한다.
- 설치 전 체크
- 설치 명령
- 설치 후 확인
- 문제 생겼을 때 중단 지점

#### 기대 결과
- 플러그인을 무작정 설치하지 않고 역할부터 파악한다.
- 설치는 "추가"가 아니라 "관리"라는 감각을 얻는다.
- 수강생은 무엇이 달라질지 알고 붙이는 습관을 갖는다.

#### 강사 설명 포인트
- "추가 기능은 많을수록 좋은 게 아니라, 필요한 것만 붙이는 게 좋습니다."
- "무엇을 왜 붙였는지 모르면 나중에 문제 원인을 찾기 어렵습니다."

### 실습 2 (15분)
#### 실습 목표
- 설치 또는 연결 과정을 단계별로 안내받는다.
- 설치 후 검증 질문을 던지는 법을 익힌다.
- 문제 발생 시 되묻는 방식까지 연습한다.

#### 따라 하기
1. 아래처럼 요청한다.

```text
gptaku 플러그인을 설치한다고 가정하고, 초보자가 실수하지 않도록 한 단계씩 안내해주세요. 각 단계마다 왜 하는지도 같이 설명해주세요.
```

2. 이어서 tofukyung도 같은 방식으로 묻는다.

```text
tofukyung 플러그인도 같은 방식으로 안내해주세요. 설치 전 확인, 설치, 설치 후 확인으로 나눠주세요.
```

3. 완료 후 검증 질문을 연습한다.

```text
설치가 끝났다고 가정하면, 정상 동작 여부를 확인하는 첫 질문 3개를 만들어주세요.
```

4. 문제가 났을 때 쓸 문장도 준비한다.

```text
설치 중 오류가 나면 제가 Claude Code에게 어떻게 질문하면 좋을지 예시 5개를 만들어주세요.
```

#### 기대 결과
- 설치 자체보다 확인 루틴이 몸에 밴다.
- 수강생은 "다 됐나요?" 대신 구체적으로 질문할 수 있다.
- 플러그인 설치도 협업 과정이라는 점을 이해한다.

#### 추천 후속 질문
- "이 플러그인은 제 업무에서 어떤 순간에 제일 먼저 써보면 좋을까요?"
- "설치가 되었는지 가장 간단하게 확인하는 방법은 무엇인가요?"
- "문제가 생기면 로그나 메시지 중 어디를 먼저 보면 되나요?"

### 마무리 (5분)
- 오늘은 플러그인을 기능 확장팩처럼 이해했다.
- 핵심은 설치 명령 외우기가 아니다.
- 역할 이해, 설치 전 확인, 설치 후 검증이라는 세 단계를 익히는 것이다.
- 김신입은 이제 새로운 도구를 붙일 때 덜 불안해질 것이다.
- 다음 레슨에서는 Git을 통해 작업 내용을 저장하고 올리는 흐름을 배운다.

#### 오늘의 한 줄 정리
- "확장은 설치보다 확인이 더 중요하다."

#### 실습 후 바로 해볼 것
1. Claude Code에게 "지금 내 환경에서 추가해도 안전한 플러그인 검토 기준"을 물어본다.
2. gptaku와 tofukyung 중 지금 업무에 더 먼저 써볼 것을 추천받는다.
3. 설치 후 점검 체크리스트를 5줄 버전으로 다시 받아본다.

## 트러블슈팅
### 증상 1: 무엇을 설치하는지 잘 모르겠다
- 해결: 먼저 역할 설명부터 듣는다.

```text
이 플러그인이 하는 일을 예시 중심으로 설명해주세요.
```

### 증상 2: 설치 단계가 너무 많아 보인다
- 해결: 단계 수를 줄여달라고 한다.

```text
초보자용 최소 단계 버전으로 다시 정리해주세요.
```

### 증상 3: 설치는 했는데 되는지 모르겠다
- 해결: 검증 질문을 요청한다.

```text
정상 동작 여부를 확인하는 가장 쉬운 테스트를 알려주세요.
```

### 증상 4: 오류 메시지가 무섭다
- 해결: 오류 전문을 보여주고 쉬운 말로 번역받는다.

```text
이 오류 메시지를 비개발자용으로 번역해주세요.
```

### 증상 5: 둘 중 무엇부터 써야 할지 모르겠다
- 해결: 현재 목표를 말하고 추천받는다.

```text
제 목표가 보고서 작성이라면 둘 중 무엇을 먼저 붙이는 게 좋을까요?
```
