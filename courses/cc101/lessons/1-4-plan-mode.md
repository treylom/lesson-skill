# 1-4: Plan 모드 "설계도 → 시공" 비유
## 학습 목표
- Plan 모드의 의미를 비개발자 관점에서 이해한다.
- 바로 실행하기보다 먼저 설계하는 습관의 장점을 배운다.
- 분석 보고서 작성 같은 업무를 단계별 계획으로 바꾸는 연습을 한다.
- Claude Code에게 실행 전에 확인받는 문화를 익힌다.

## 사전 준비
- 1-3 레슨까지 완료
- `CLAUDE.md`의 역할을 이해한 상태
- `practice-data/` 폴더를 한 번 둘러본 상태
- "생각 없이 바로 시작하면 나중에 수정 비용이 커진다"는 경험에 공감할 수 있는 상태
- 짧은 계획표를 읽고 판단할 준비

## 진행
### 도입 (5분)
- 많은 초보자는 AI에게 바로 결과를 시키고 싶어 한다.
- 하지만 일은 보통 "설계 → 실행 → 검토" 순서로 할 때 덜 흔들린다.
- 집을 지을 때도 설계도 없이 바로 벽돌부터 쌓지 않는다.
- Plan 모드는 Claude Code에게 **먼저 설계도를 그리게 하는 시간**이다.
- 즉, 지금 당장 파일을 바꾸거나 결과물을 만들기 전에,
- 무엇을 할지,
- 어떤 순서로 할지,
- 무엇을 확인해야 할지 정리하는 단계다.
- 김신입에게 이 기능은 특히 중요하다.
- 아직 익숙하지 않은 업무일수록 "먼저 계획을 보여줘"가 큰 안전장치가 되기 때문이다.

#### 쉬운 비유
- 일반 요청: "바로 공사 시작해"
- Plan 모드: "먼저 설계도와 공정표부터 보여줘"
- 초보자에게는 두 번째가 훨씬 안전하다.

#### 체크 포인트
- 수강생이 계획과 실행의 차이를 이해하는가?
- AI의 첫 답을 그대로 실행하기보다 검토할 수 있다는 점을 이해하는가?
- 본인도 계획을 보고 수정 의견을 줄 수 있다는 사실을 인식하는가?

### 실습 1 (15분)
#### 실습 목표
- Plan 모드가 필요한 상황을 구분한다.
- 짧은 업무를 계획표 형태로 받아본다.
- 계획의 품질을 보는 기준을 익힌다.

#### 따라 하기
1. Claude Code에게 아래처럼 요청한다.

```text
practice-data를 바탕으로 첫 분석 보고서를 만들고 싶어요. 바로 작성하지 말고, 먼저 어떤 순서로 진행하면 좋을지 계획부터 보여주세요.
```

2. 답변이 오면 아래 기준으로 읽는다.
- 단계가 너무 크지 않은가?
- 데이터 읽기와 보고서 작성이 구분되어 있는가?
- 검토 단계가 있는가?
- 결과물을 무엇으로 만들지 적혀 있는가?

3. 이어서 더 구체화한다.

```text
좋아요. 이 계획을 비개발자도 따라할 수 있게 6단계 이내로 다시 정리해주세요. 각 단계마다 제가 확인해야 할 포인트도 넣어주세요.
```

4. 계획표를 읽고 마음에 들지 않는 부분을 수정 요청한다.

```text
3단계와 4단계가 비슷해 보여요. 더 단순하게 합쳐주세요.
```

#### 기대 결과
- 수강생은 계획도 AI와 함께 다듬는 대상임을 알게 된다.
- "좋은 계획은 내가 이해할 수 있는 계획"이라는 기준이 생긴다.
- Claude Code가 단순 작성기가 아니라 기획 파트너처럼 느껴진다.

#### 계획을 볼 때의 질문
- 이 순서라면 내가 따라갈 수 있는가?
- 각 단계의 끝에서 무엇이 남는가?
- 지금 당장 하지 않아도 되는 일이 섞여 있지 않은가?

### 실습 2 (15분)
#### 실습 목표
- Plan 모드를 실제 업무 상황에 적용한다.
- 보고서 작성과 GitHub 정리까지 포함한 작은 프로젝트 계획을 받는다.
- 실행 전에 승인하는 습관을 만든다.

#### 따라 하기
1. 아래처럼 더 큰 범위의 계획을 요청한다.

```text
저는 제조업 회사의 신입 AI 담당자예요. practice-data를 분석해서 보고서.md를 만들고, 결과를 GitHub 저장소로 정리하고 싶어요. 지금은 실행하지 말고, 먼저 설계도처럼 전체 계획을 보여주세요.
```

2. 답변이 오면 확인 질문을 던진다.

```text
이 계획에서 가장 먼저 해야 할 한 가지와, 마지막에 확인해야 할 한 가지를 알려주세요.
```

3. 계획이 너무 어렵다면 난이도를 낮춘다.

```text
좋아요. 이걸 컴퓨터가 익숙하지 않은 신입 사원도 따라할 수 있게 더 쉬운 표현으로 다시 써주세요.
```

4. 마지막으로 승인 문화를 연습한다.

```text
이제 제가 검토할 수 있도록, 계획의 핵심만 표 형태로 요약해주세요. 제가 승인하면 그다음 단계로 넘어가는 방식으로 진행해주세요.
```

#### 기대 결과
- 수강생은 "바로 해줘" 대신 "먼저 계획 보여줘"를 습관화한다.
- 작업 범위가 클수록 설계도가 더 중요하다는 점을 이해한다.
- AI를 통제하는 감각이 생긴다.

#### 강사 설명 포인트
- "초보자일수록 계획을 보고 들어가는 편이 덜 무섭습니다."
- "계획을 검토하는 순간, AI에 끌려가지 않고 내가 운전대를 잡게 됩니다."

#### 계획 품질 체크리스트
- 시작 단계가 분명한가?
- 중간 산출물이 보이는가?
- 마지막 결과물이 구체적인가?
- 내가 이해할 수 있는 단어로 적혀 있는가?
- 지금 당장 필요 없는 일이 빠져 있는가?

### 마무리 (5분)
- 오늘 배운 핵심은 간단하다.
- **설계도 없이 바로 시공하지 않는다.**
- Claude Code를 잘 쓰는 사람은 AI에게 바로 결과만 시키지 않는다.
- 먼저 계획을 받는다.
- 그 계획을 읽고 고친다.
- 그리고 나서 실행한다.
- 김신입은 이제 파일을 읽고, 맥락을 주고, 계획까지 세울 수 있게 되었다.
- 다음 모듈에서는 플러그인, Git, 공식 문서 활용을 통해 실무 도구를 넓힌다.

#### 오늘의 한 줄 정리
- "좋은 실행은 좋은 설계도에서 시작된다."

#### 실습 후 바로 해볼 것
1. Claude Code에게 "제가 자주 하는 업무 중 계획부터 세우면 좋은 일 3가지를 골라주세요"라고 묻는다.
2. 오늘 받은 계획을 4단계 버전으로 줄여달라고 한다.
3. "제가 승인하기 전에는 실행하지 마세요"라는 문장을 붙여본다.

## 트러블슈팅
### 증상 1: 계획이 너무 추상적이다
- 해결: 결과물과 단계를 더 구체적으로 요구한다.

```text
각 단계 끝에 무엇이 완성되는지 적어주세요.
```

### 증상 2: 계획이 너무 길다
- 해결: 제한을 건다.

```text
6단계 이내로 줄여주세요.
```

### 증상 3: 아직도 바로 결과를 받고 싶다
- 해결: 큰 작업일수록 수정 비용이 더 크다는 점을 떠올린다.
- 먼저 계획을 보면 나중에 덜 돌아간다.

### 증상 4: 내가 계획을 검토할 자신이 없다
- 해결: 아래처럼 평가 기준도 함께 요청한다.

```text
이 계획을 초보자가 검토할 때 볼 포인트를 함께 써주세요.
```

### 증상 5: Plan 모드와 일반 질문의 차이가 잘 안 느껴진다
- 해결: 같은 주제로 두 방식 모두 시도해보고 비교한다.
- "바로 보고서 써줘"와 "먼저 계획 보여줘"의 차이를 체감한다.
