# 2-3: "모르면 공식 문서에 물어봐" — Docs 가이드
## 학습 목표
- 모르는 기능을 추측하지 않고 공식 문서를 찾는 습관을 만든다.
- Claude Code를 문서 통역사처럼 활용하는 방법을 배운다.
- 설명, 비교, 예시 요청을 통해 공식 문서를 실무 언어로 바꾸는 법을 익힌다.
- 비개발자도 "문서 기반 질문"을 할 수 있다는 자신감을 얻는다.

## 사전 준비
- 2-2 레슨까지 완료
- 인터넷 연결 가능 상태
- 모르는 기능이 나와도 바로 포기하지 않을 준비
- 문서는 읽는 것이 아니라 "찾고 물어보는 것"부터 시작한다는 이해
- 한 번에 다 이해하지 않아도 괜찮다는 합의

## 진행
### 도입 (5분)
- 초보자는 종종 이렇게 생각한다.
- "문서는 개발자만 보는 것 아닌가요?"
- 하지만 실무에서는 반대다.
- 모를 때 추측하는 사람이 아니라, 근거를 확인하는 사람이 더 빠르게 성장한다.
- 공식 문서는 제품이 자기 기능을 가장 정확하게 설명한 자료다.
- 다만 문제는 문서가 어렵게 느껴진다는 것이다.
- 여기서 Claude Code의 역할이 나온다.
- Claude Code는 공식 문서를 대신 읽고,
- 중요한 부분을 추리고,
- 내 상황에 맞는 말로 바꿔줄 수 있다.
- 오늘 목표는 문서를 처음부터 끝까지 읽는 것이 아니다.
- **필요한 질문을 던져 필요한 답만 꺼내는 것**이다.

#### 쉬운 비유
- 공식 문서는 제품 설명서다.
- Claude Code는 그 설명서를 옆에서 풀어주는 통역사다.

#### 체크 포인트
- 수강생이 추측보다 확인이 중요하다는 데 동의하는가?
- 문서를 한 번에 다 이해하려는 부담을 내려놓았는가?
- 필요한 부분만 묻는 방식이 가능하다는 걸 이해했는가?

### 실습 1 (15분)
#### 실습 목표
- 공식 문서를 읽는 대신, Claude Code에게 문서 질문을 던지는 연습을 한다.
- 기능 설명을 쉬운 말로 바꿔달라고 요청한다.
- 내 상황에 맞는 사용 예시를 받는다.

#### 따라 하기
1. Claude Code에게 아래처럼 요청한다.

```text
제가 Claude Code 초보자인데, 공식 문서를 볼 때 무엇부터 확인하면 좋은지 비개발자용 체크리스트를 만들어주세요.
```

2. 이어서 기능 하나를 고른다.
- 예: Plan 모드
- 예: CLAUDE.md
- 예: GitHub 관련 도움

3. 아래처럼 묻는다.

```text
공식 문서를 기준으로, Plan 모드가 무엇이고 언제 쓰는지 초보자 눈높이로 설명해주세요.
```

4. 더 실무적으로 좁힌다.

```text
제 상황이 제조업 회사 신입 AI 담당자라면, 이 기능을 언제 가장 먼저 써보면 좋은지 예시 3개를 들어주세요.
```

#### 기대 결과
- 문서 질문은 추상적일 필요가 없다는 걸 배운다.
- 기능 설명 + 언제 쓰는지 + 내 상황 예시를 함께 받으면 이해가 훨씬 빨라진다.
- 수강생은 문서를 직접 다 읽지 않아도 핵심을 끌어낼 수 있음을 체감한다.

#### 강사 설명 포인트
- "문서를 읽는 능력보다 중요한 것은, 문서에 무엇을 물어볼지 아는 능력입니다."
- "모르면 창피한 것이 아니라, 추측하고 틀리는 것이 더 비효율적입니다."

### 실습 2 (15분)
#### 실습 목표
- 공식 문서를 바탕으로 비교 질문과 실전 질문을 던진다.
- 어려운 용어를 쉬운 말로 재설명받는다.
- "문서를 읽은 뒤 무엇을 해야 하는지"까지 연결한다.

#### 따라 하기
1. 아래처럼 비교 질문을 한다.

```text
공식 문서를 기준으로, 일반 요청과 Plan 모드의 차이를 표로 설명해주세요.
```

2. 이어서 용어 설명을 요청한다.

```text
문서에 나오는 핵심 용어를 중학생도 이해할 수 있게 풀어주세요.
```

3. 실전 적용 질문을 던진다.

```text
제가 practice-data 분석 보고서를 만들려는 상황이라면, 공식 문서를 기준으로 어떤 기능을 어떤 순서로 쓰면 좋을지 추천해주세요.
```

4. 마지막으로 문서 기반 행동 계획을 받는다.

```text
좋아요. 그 내용을 바탕으로 오늘 바로 해볼 수 있는 20분짜리 실습 루틴을 만들어주세요.
```

#### 기대 결과
- 수강생은 문서를 단순 정보가 아니라 행동 근거로 사용하게 된다.
- 비교 질문, 쉬운 말 번역, 내 상황 적용이라는 3단계 패턴을 익힌다.
- "모르면 공식 문서에 물어봐"가 실제 행동 습관으로 바뀐다.

#### 좋은 질문 패턴
- 공식 문서를 기준으로 설명해주세요.
- 초보자 눈높이로 다시 말해주세요.
- 제 상황에 맞는 예시를 들어주세요.
- 바로 해볼 수 있는 단계로 바꿔주세요.

### 마무리 (5분)
- 오늘 배운 것은 아주 실무적이다.
- 모를 때 감으로 때우지 않는다.
- 공식 문서를 근거로 삼고, Claude Code에게 통역을 맡긴다.
- 이 습관이 생기면 새로운 도구가 나와도 덜 무섭다.
- 김신입은 이제 모를 때 얼어붙는 사람이 아니라,
- 근거를 찾아 행동으로 연결하는 사람이 된다.
- 다음 시간에는 지금까지 배운 것을 합쳐 첫 작은 프로젝트를 끝낸다.

#### 오늘의 한 줄 정리
- "추측보다 문서, 문서보다 적용이 중요하다."

#### 실습 후 바로 해볼 것
1. Claude Code에게 "초보자가 공식 문서에서 우선 확인할 항목 5개"를 다시 요청한다.
2. 본인이 헷갈린 기능 하나를 골라 문서 기반 설명을 받아본다.
3. 그 설명을 바탕으로 10분짜리 실습 단계까지 받아본다.

## 트러블슈팅
### 증상 1: 문서라는 말만 들어도 부담스럽다
- 해결: 전체 문서가 아니라 필요한 질문 한 개만 던진다.

```text
이 기능이 언제 필요한지만 알려주세요.
```

### 증상 2: 답변이 여전히 어렵다
- 해결: 난이도를 더 낮춘다.

```text
중학생도 이해할 수 있게 다시 설명해주세요.
```

### 증상 3: 문서 내용을 읽었는데 실무 연결이 안 된다
- 해결: 내 상황을 붙인다.

```text
제 업무 상황에 맞춰 이 기능의 사용 예시를 알려주세요.
```

### 증상 4: 어떤 문서를 봐야 할지 모르겠다
- 해결: 먼저 질문한다.

```text
제 목표가 보고서 작성이라면 어떤 공식 문서를 우선 보면 좋을까요?
```

### 증상 5: 공식 문서와 인터넷 글이 달라 보인다
- 해결: 공식 문서를 우선 기준으로 삼는다.
- 블로그나 후기 글은 보조 참고로만 쓴다.
