# 2-2: Git 기초 — 바르다깃선생 "저장해줘" + "올려줘"
## 학습 목표
- Git을 비개발자 기준으로 이해한다.
- "저장해줘"와 "올려줘"의 차이를 배운다.
- 로컬 저장과 원격 업로드를 구분한다.
- Git이 무섭기만 한 도구가 아니라 작업 이력을 남기는 안전장치라는 것을 체감한다.

## 사전 준비
- 작업 폴더가 하나 준비되어 있음
- 보고서나 메모처럼 저장할 파일이 최소 1개 있음
- 인터넷 연결 가능 상태면 더 좋음
- GitHub 계정이 있으면 좋지만 없어도 학습 가능
- 명령어보다 개념부터 이해하겠다는 자세

## 진행
### 도입 (5분)
- 많은 비개발자는 Git을 "개발자만 쓰는 무서운 것"으로 본다.
- 하지만 더 쉬운 비유가 있다.
- Git은 **문서 작업의 저장 이력 관리 도구**다.
- 여기서 두 가지를 구분하면 훨씬 쉽다.
- 첫 번째는 "저장해줘"다.
- 이것은 지금까지의 작업 상태를 로컬 컴퓨터 안에 기록으로 남기는 것이다.
- 두 번째는 "올려줘"다.
- 이것은 그 기록을 GitHub 같은 원격 저장소에 보내는 것이다.
- 즉, 저장과 업로드는 다르다.
- 바르다깃선생 방식으로 배우면 이렇게 기억하면 된다.
- 저장해줘 = commit
- 올려줘 = push
- 오늘 목표는 이 두 문장의 차이를 몸으로 이해하는 것이다.

#### 쉬운 비유
- 워드 파일 저장: 내 컴퓨터에 저장
- 드라이브 공유: 다른 곳에도 올림
- Git은 이 과정을 더 체계적으로 남긴다.

#### 체크 포인트
- 로컬과 원격의 차이를 이해하는가?
- 기록을 남기는 것이 왜 중요한지 공감하는가?
- 잘못 저장해도 다시 볼 수 있다는 점이 안심되는가?

### 실습 1 (15분)
#### 실습 목표
- Git의 개념을 Claude Code에게 쉬운 말로 설명받는다.
- "저장"과 "업로드"를 구분하는 연습을 한다.
- 김신입의 보고서 폴더를 예시로 설명을 받아본다.

#### 따라 하기
1. Claude Code에게 아래처럼 질문한다.

```text
Git을 비개발자에게 설명해주세요. 특히 "저장해줘"와 "올려줘"의 차이를 제 보고서 폴더 예시로 설명해주세요.
```

2. 답변을 읽고 이어서 묻는다.

```text
제가 보고서.md를 수정한 뒤, 어떤 순서로 기록을 남기고 나중에 GitHub에 올리게 되는지 5단계로 설명해주세요.
```

3. 이제 용어 번역을 요청한다.

```text
init, add, commit, push를 각각 "사람말"로 번역해주세요.
```

4. 메모장에 본인식 표현으로 적어본다.
- init = 저장 창고 만들기
- add = 이번에 저장할 것 고르기
- commit = 저장 이름 붙이기
- push = 원격으로 올리기

#### 기대 결과
- Git이 개발자 마법이 아니라 저장 이력 도구라는 이해를 얻는다.
- 용어를 본인 언어로 바꾸면 두려움이 크게 줄어든다.
- 저장과 올리기의 순서를 기억하게 된다.

#### 강사 설명 포인트
- "Git은 실수하지 않게 해주는 보험에 가깝습니다."
- "가장 중요한 건 명령어 암기보다 흐름 이해입니다."

### 실습 2 (15분)
#### 실습 목표
- 실제 Git 작업 흐름을 말로 따라간다.
- Claude Code에게 각 단계에서 무슨 일이 일어나는지 설명받는다.
- GitHub 업로드까지의 전체 그림을 만든다.

#### 따라 하기
1. 아래처럼 요청한다.

```text
제 작업 폴더를 Git으로 관리한다고 가정하고, 완전 초보자용으로 init → add → commit → push 흐름을 한 단계씩 설명해주세요. 각 단계마다 제가 확인할 포인트도 알려주세요.
```

2. 이어서 상황형 질문을 한다.

```text
제가 보고서.md 파일 하나만 바꿨다면, "저장해줘"와 "올려줘"는 각각 어느 단계인지 다시 설명해주세요.
```

3. 마지막으로 실수 대비 질문을 한다.

```text
제가 헷갈리기 쉬운 실수 5가지와, 그럴 때 Claude Code에게 어떻게 물어보면 좋은지 예시를 알려주세요.
```

4. 가능하다면 GitHub 연결 맥락도 묻는다.

```text
GitHub는 왜 쓰는지, 제 혼자 작업에도 왜 도움이 되는지 설명해주세요.
```

#### 기대 결과
- 수강생은 각 단계의 목적을 이해한다.
- "commit은 저장 기록", "push는 외부 공유 또는 백업"이라는 개념이 분명해진다.
- GitHub가 협업뿐 아니라 개인 작업 정리에도 유용함을 이해한다.

#### 좋은 후속 질문
- "지금 제 폴더 상태를 확인하는 가장 쉬운 방법은 무엇인가요?"
- "커밋 메시지는 어떤 식으로 쓰면 좋나요?"
- "저장만 하고 아직 올리지 않은 상태는 어떤 의미인가요?"

### 마무리 (5분)
- 오늘은 Git을 처음 만났다.
- 핵심은 네 글자 암기가 아니다.
- "저장해줘"와 "올려줘"를 구분하는 것이다.
- 김신입은 이제 보고서 작업을 그냥 덮어쓰는 사람이 아니라,
- 기록을 남기고 필요하면 공유할 수 있는 사람이 된다.
- 다음 레슨에서는 모를 때 공식 문서에 물어보는 습관을 배운다.

#### 오늘의 한 줄 정리
- "저장은 내 기록, 올리기는 내 공유다."

#### 실습 후 바로 해볼 것
1. Claude Code에게 "초보자가 Git에서 제일 먼저 익혀야 할 명령 3개"를 묻는다.
2. commit과 push의 차이를 본인 말로 한 줄씩 정리해본다.
3. 내 보고서 작업을 Git으로 관리하면 좋은 이유를 3가지 써본다.

## 트러블슈팅
### 증상 1: 용어가 너무 어렵다
- 해결: 사람말 번역을 계속 요청한다.

```text
commit을 메신저/문서 작업 비유로 설명해주세요.
```

### 증상 2: 로컬과 GitHub가 헷갈린다
- 해결: 비교표로 설명받는다.

```text
내 컴퓨터와 GitHub의 차이를 표로 설명해주세요.
```

### 증상 3: 저장은 했는데 올리진 않은 상태가 불안하다
- 해결: 아래처럼 상태 설명을 요청한다.

```text
저장만 하고 아직 올리지 않은 상태가 어떤 의미인지 설명해주세요.
```

### 증상 4: 명령 순서가 자꾸 섞인다
- 해결: 구호처럼 외운다.
- 준비한다 → 고른다 → 저장한다 → 올린다

### 증상 5: Git이 괜히 무서워진다
- 해결: Claude Code에게 초보자용 최소 흐름만 다시 받는다.

```text
초보자용 최소 Git 흐름만 다시 알려주세요.
```
